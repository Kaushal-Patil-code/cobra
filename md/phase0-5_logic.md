# COBRA — Phase 0 to 5 Logic (Pointwise)

> The actual implemented logic of phases 0–5, read from the source code (not just the spec).
> Each phase is a single flat list of standalone points — no sub-points, no nesting.
> Spec reference: `data_md/oi_confirmation_dashboard_spec.md`. Phases 6–8 are out of scope here.

## Phase 0 — Fyers daily auth (headless) · [spec §7]

Files: `auth/auth.py`, `auth/force_login.py`, `auth/startup.py`, `auth/__init__.py`, `db/db.py`, `config/settings.py`

1. The six Fyers credentials (`fyers_app_id`, `fyers_secret_key`, `fyers_redirect_url`, `fyers_username`, `fyers_pin`, `fyers_totp_secret`) are read from environment/`.env` via the pydantic-settings `Settings` singleton in `config/settings.py`, all typed `Optional[str]` defaulting to `None`, and loaded lazily inside `auth.py`'s `_get_credentials()` rather than at import time so secrets are not read into memory until login is actually attempted.
2. `get_fyers_token()` is the main entry point and tries three strategies strictly in order — (1) reuse the cached token if fresh, (2) call the Fyers refresh API using the stored refresh_token, (3) run the full 5-step TOTP login with flow-level retry — returning an empty string only if all three fail.
3. Cached-token reuse is gated by `MAX_TOKEN_AGE_HOURS = 22` (a 2-hour safety margin on Fyers' ~24h token lifetime): `age_hours = (now_IST - saved_timestamp)` in seconds / 3600, and the cached `access_token` is returned only when `age_hours < 22`, otherwise the token is logged expired and the flow falls through.
4. All timestamp math uses `IST = ZoneInfo("Asia/Kolkata")`, comparing `datetime.now(IST)` against the ISO-8601 timestamp via `datetime.fromisoformat()`; a `ValueError`/`TypeError` parsing the cached timestamp is caught and treated as a cache miss that proceeds to re-login.
5. Tokens are persisted in the Postgres `fyers_tokens` table (one row per `client_id`, keyed on `settings.fyers_app_id`), not on disk, with columns `access_token`, `refresh_token`, `updated_at`, where `updated_at` doubles as the issued-at timestamp used for the freshness check.
6. `save_token()` writes with a plain `UPDATE` keyed on `client_id` and only falls back to an `INSERT` when `cur.rowcount == 0` (no existing row on first-ever login), deliberately avoiding an upsert, and stores `datetime.now(IST)` as `updated_at`.
7. `load_token()` SELECTs `access_token`, `refresh_token`, `updated_at` for the configured `client_id` and returns a legacy-file-shaped dict `{access_token, refresh_token, timestamp(ISO)}`; it returns `{}` when `client_id` is empty, when no row exists, or on any DB exception (table missing, DB unreachable), so `get_fyers_token()` degrades to refresh/full-login exactly as a missing file would.
8. Strategy 2 `refresh_access_token()` POSTs to `https://api-t1.fyers.in/api/v3/validate-refresh-token` with `grant_type="refresh_token"`, `appIdHash`, the `refresh_token`, and the PIN as a string, skipping all OTP/TOTP/PIN-verify/auth-code steps; on success it returns the new `access_token` (keeping the old `refresh_token` if the response omits a new one) and returns `{}` on any network/parse error, which falls through to full login.
9. `appIdHash` is computed by `_app_id_hash()` as `sha256("{app_id}:{secret_key}").hexdigest()` and is required by both the `validate-refresh-token` and `validate-authcode` direct-API endpoints.
10. The full TOTP flow is five sequential POST steps where each step's output feeds the next, aborting (returning `{}`) immediately if any step yields an empty result: Step 1 `send_login_otp` → `request_key`, Step 2 `verify_otp` → new `request_key`, Step 3 `verify_pin` → temporary `access_token`, Step 4 `token` → `auth_code`, Step 5 `validate-authcode` → final `access_token` + `refresh_token`.
11. Step 1 (`step1_send_login_otp`) POSTs to `https://api-t2.fyers.in/vagator/v2/send_login_otp` with payload `{"fy_id": username, "app_id": "2"}` (`app_id` hardcoded to `"2"` for web/API login) and returns `d["request_key"]` when `d["s"] == "ok"`.
12. Step 2 (`step2_verify_totp`) generates the current 6-digit code with `pyotp.TOTP(totp_secret).now()` and POSTs it to `https://api-t2.fyers.in/vagator/v2/verify_otp` as a STRING (preserving leading zeros like `"063265"`), returning the new `request_key` on success.
13. Step 3 (`step3_verify_pin`) POSTs to `https://api-t2.fyers.in/vagator/v2/verify_pin` with `{"request_key", "identity_type": "pin", "identifier": str(pin)}` and extracts the temporary token from `d["data"]["access_token"]`.
14. Step 4 (`step4_get_auth_response`) POSTs to `https://api-t1.fyers.in/api/v3/token` with an `Authorization: Bearer <temp_token>` header and a payload that strips the app_id suffix via `app_id.split("-")[0]`, sets `appType="100"`, `response_type="code"`, `state="sample_state"`, `create_cookie=True`, empty `code_challenge`/`scope`/`nonce`; it reads the redirect URL from the capital-U `"Url"` field and extracts `auth_code` via `parse_qs(urlparse(url).query).get("auth_code", [""])[0]`, failing with `ok=False` if status is non-ok or `auth_code` is missing.
15. `step4_get_auth_code()` is a thin wrapper returning only the bare `auth_code` string (used by `full_totp_login`), while `get_auth_code()` runs steps 1–4 only (deliberately NOT step 5) and returns the full structured response (url, cookie, code, status, raw) with a per-step error/step indicator.
16. Step 5 (`step5_validate_auth_code`) first tries Method A — the official `fyersModel.SessionModel.generate_token()` SDK call wrapped in a `ThreadPoolExecutor` with a 15-second `future.result(timeout=15)` because the SDK has no built-in timeout and can hang — accepting success when the SDK result has `s == "ok"` OR `code == 200`; on `FuturesTimeoutError` or any exception it falls back to Method B, a direct POST to `https://api-t1.fyers.in/api/v3/validate-authcode` with `grant_type="authorization_code"`, `appIdHash`, and the code, which accepts success only on `s == "ok"` (Method B has no `code == 200` branch).
17. Per-request network resilience is provided by `_post_with_retry()` which retries up to `MAX_RETRIES = 5` times with `RETRY_DELAY = 2` seconds and a default 10-second timeout, retrying ONLY on `requests.ConnectionError`/`requests.Timeout` while API-level errors (4xx, wrong credentials, non-ok responses) are permanent and not retried.
18. `_post_with_retry()` re-raises the last `ConnectionError`/`Timeout` via `raise last_exc` when retries are exhausted (it does not return `{}`), which is why every step/refresh wraps the call in `try/except requests.RequestException` to convert that into an empty-result failure.
19. `_post_with_retry()` returns `resp.json()` (raising `ValueError` if the body is not valid JSON, which callers catch alongside `KeyError`), and any non-`ConnectionError`/non-`Timeout` exception from `requests.post` is not retried at all.
20. Flow-level retry is layered on top of per-request retry: `full_totp_login_with_retry()` re-runs the entire 5-step `full_totp_login()` up to `fyers_login_max_attempts` times (default 3, `ge=1`, floored with `max(1, int(max_attempts))`) sleeping `fyers_login_retry_delay` seconds (default 5, `ge=0`) between attempts, each retry using a fresh `request_key` and fresh TOTP code to survive a 30-second TOTP window boundary, but treats missing credentials as a permanent error returning `{}` immediately with no retry.
21. `full_totp_login()`, `full_totp_login_with_retry()`, and `get_auth_code()` first validate that all six credentials are present (`missing = [k for k,v in creds.items() if not v]`) and abort early with `{}` (logging which keys are missing in `.env`) before making any API call.
22. Startup warm-up (`warm_login_once` in `auth/startup.py`) is single-flight across workers via a Postgres transaction-level advisory lock `pg_try_advisory_xact_lock(_LOGIN_LOCK_KEY)` where `_LOGIN_LOCK_KEY = 728_051_193`; only the worker that wins the non-blocking try-lock proceeds, others log and skip, the lock auto-releases on commit/rollback/crash (pooler-safe), it no-ops when credentials are missing, and it never raises (all Fyers/DB errors are caught and logged so boot is not blocked).
23. `warm_login_once()` calls `ensure_valid_token()` (not the cheaper `get_fyers_token()`) so the startup token is validated live; the `fyers_autologin_on_startup` flag (default True) deciding whether warm-up runs at all is checked by the caller in `app/__init__.py`, not inside `warm_login_once()` itself, whose only gates are the missing-credentials no-op and the advisory try-lock.
24. `ensure_valid_token()` does a live validation by making one `client.get_profile()` call on the token from `get_fyers_token()` and returns it only if the response is a dict with `s == "ok"`; otherwise it calls `invalidate_token()` and re-runs `get_fyers_token()`, because the 22h age heuristic can consider a token fresh even though Fyers expired it on a daily cutoff.
25. `invalidate_token()` marks a token stale by `UPDATE fyers_tokens SET updated_at = to_timestamp(0)` (the epoch) for the `client_id` while preserving the `refresh_token`, so the next `get_fyers_token()` fails the freshness check and tries refresh first, then full TOTP login.
26. `relogin_single_flight()` (reactive re-auth when a live call rejects the token) takes a BLOCKING `pg_advisory_xact_lock` on the same `_LOGIN_LOCK_KEY`, snapshots the token timestamp before waiting (calling `load_token()` once before the lock and once after), and after acquiring the lock reuses the token if another worker already refreshed it (changed timestamp + present access_token); only if the timestamp is unchanged does it `invalidate_token()` and run `get_fyers_token()`, returning a fresh `FyersModel` client on both paths and raising `RuntimeError` if no token is ultimately obtained.
27. `force_login.force_full_login()` bypasses `load_token()`/refresh entirely and always runs steps 1–5 via `full_totp_login()`, serializing concurrent callers with a module-level `threading.Lock` so the TOTP flow never runs twice in parallel (avoiding double API charges and Fyers anti-fraud throttling), and persists the result via `save_token()` on success.
28. All Fyers HTTP requests use a shared `HEADERS` dict that mimics Chrome (User-Agent `Mozilla/5.0`, Origin `https://api-t1.fyers.in`, Referer `https://api-t1.fyers.in/`) to satisfy Fyers' CORS/Origin checks, and `get_fyers_client()` builds a synchronous `fyersModel.FyersModel(client_id=app_id, token=token, is_async=False)`, raising `RuntimeError("No valid Fyers token. Check .env")` when `get_fyers_token()` returns an empty string.
29. `auth.py` exposes a public `get_profile()` helper (exported in `auth/__init__.py`) that builds a client via `get_fyers_client()`, calls `client.get_profile()`, returns `resp["data"]` when `resp["s"] == "ok"`, and raises `RuntimeError(f"Profile fetch failed: {resp}")` otherwise.
30. `db.py` provides the `get_conn()` context manager used by all token DB operations: the transaction is committed on clean block exit and rolled back if an exception propagates, and the pool is built lazily from `settings.effective_dsn`, raising `RuntimeError` if no DSN is configured.
31. `db.py` disables server-side prepared statements when the Supabase transaction-mode pooler is detected (`_is_pooler`: `":6543"` in dsn or `"pgbouncer=true"`) or when `settings.db_disable_prepared_statements` is set, and skips the per-connection `statement_timeout` SET on the pooler because a session SET would not persist across pooled transactions.
32. `settings.effective_dsn` resolves the connection string by returning `database_url` if set, otherwise assembling `postgresql://user:pass@host:port/name?sslmode=...` from the `DB_*` parts (URL-encoding the password via `quote`), and returns `None` if neither is available.
33. `config/settings.py` is built on pydantic-settings `BaseSettings` with `env_file='.env'`, `case_sensitive=False`, `extra='ignore'` (to tolerate unrelated env keys), and the singleton is created once via an `lru_cache(maxsize=1)` `get_settings()` factory so validation runs once at import.

## Phase 1 — Fetcher + raw snapshots · [spec §2]

Files: `market/chain.py`, `market/fetch.py`, `market/parse.py`, `market/store.py`, `market/instruments.py`, `schemas/market.py`

1. `fetch_raw_chain()` in `market/chain.py` issues exactly one Fyers `client.optionchain()` call per underlying with `data={"symbol": symbol, "strikecount": strikecount, "timestamp": timestamp}`, so a full cycle over both indices is two calls (spec §2).
2. The `optionchain` call defaults to `strikecount=10` and `timestamp=""`, where the empty timestamp selects the nearest expiry's chain while the response still lists every expiry in `expiryData`.
3. `fetch_raw_chain()` retries transient failures up to `FETCH_MAX_ATTEMPTS=3` times with `FETCH_RETRY_DELAY=2` seconds between attempts, sleeping only when there are remaining attempts.
4. A pull is considered successful only when the response is a dict with `resp.get("s") == "ok"`; any non-ok dict is wrapped as a `ChainFetchError` and retried.
5. `_is_token_error()` detects an auth failure when the response dict has `code == -15` or its message contains the substring `"valid token"` (case-insensitive), and such a response immediately raises `TokenRejectedError` without retrying because a dead token cannot be fixed by retrying.
6. Network or SDK exceptions raised by `client.optionchain()` are caught and treated as transient (retried), while exhausting all attempts raises `ChainFetchError` so partial data is never stored.
7. `fetch_chains()` in `market/fetch.py` computes one shared `fetched_at = datetime.now(timezone.utc)` for the entire cycle and assigns it to every parsed `ChainSnapshot` so all rows of a cycle align on one timestamp.
8. `fetch_chains()` iterates over `all_instruments().values()` and is per-index resilient (spec §11): a non-token `Exception` on one index is logged and that index is skipped via `break`, while the others still proceed.
9. On a `TokenRejectedError`, `fetch_chains()` calls `_relogin()` (`auth.startup.relogin_single_flight`, single-flight across workers) once, sets `reauthed=True`, and retries that index with the fresh client via an inner `range(2)` loop.
10. If re-authentication itself raises, `fetch_chains()` logs the error and gracefully returns whatever chains were fetched so far (possibly empty) rather than raising, and if the token is still rejected after one re-auth it logs and skips that index.
11. `run_fetch()` first obtains the client via `get_fyers_client()` (cached token → refresh → full TOTP login) and returns `{"error": "auth: ..."}` without fetching if auth fails.
12. `run_fetch()` takes `fetched_at` from `next(iter(chains.values())).fetched_at` when any chains exist (None otherwise), seeds a top-level `fetched_at` key as `fetched_at.isoformat()` (or None), and builds a per-index summary with rows, distinct strike count, expiry, vix, call_oi, and put_oi.
13. `run_fetch()` wraps each `store_snapshot()` call in `try/except` so a store failure for one index is logged and recorded as `{"error": ..., "rows": 0}` without aborting the others.
14. `parse_chain()` in `market/parse.py` treats `data.optionsChain[0]` (the underlying row with `strike_price -1` and blank `option_type`) and any row whose `option_type` is not in `("CE", "PE")` as skippable, keeping only CE/PE strike rows.
15. `_parse_expiries()` raises `ValueError` if `expiryData` is missing or empty, builds each `ExpiryInfo` with `epoch=str(e.get("expiry") or "")` and `flag=e.get("expiry_flag")` (carrying the per-expiry weekly/monthly flag), and `parse_chain()` takes `available[0]` as the nearest expiry because `expiryData` is ascending.
16. `parse_chain()` exposes the nearest expiry as both `expiry` and `expiry_epoch` on the `ChainSnapshot`, where `expiry_epoch = nearest.epoch or None` (an empty-string epoch is coerced to None), and also attaches the full `available_expiries` list so later phases can re-query a specific expiry by epoch.
17. `_expiry_date()` prefers the integer `expiry` epoch (resolved to an IST date via `ZoneInfo("Asia/Kolkata")`) and only falls back to parsing the `date` string against formats `"%d-%m-%Y"` then `"%d-%b-%Y"`, raising `ValueError` if nothing parses.
18. India VIX is read from `data.indiavixData.ltp` and applied to every chain including SENSEX, because Fyers returns NSE India VIX even on the Sensex chain and there is no Sensex-specific volatility index.
19. Strike intervals come from the constant `STRIKE_INTERVAL = {"NIFTY": 50, "SENSEX": 100}`, and any strike not divisible by that interval is logged as a warning but still kept (kept-but-flagged).
20. Each `StrikeOI` captures `index_name`, `option_type`, `strike` (int), `expiry`, `oi` (int, defaulting to 0 when `oi` is None), `ltp` (stored raw from `row.get("ltp")` without `_as_int`, kept float/None), `volume`, `prev_oi`, and `oichp`, where `_as_int()` converts `volume`/`prev_oi` to int or None.
21. Chain totals `call_oi`/`put_oi` are chain-level fields on `ChainSnapshot` (read from camelCase `data.get("callOi")`/`data.get("putOi")` via `_as_int`), distinct from per-strike `oi`, and are NOT fields of `StrikeOI`.
22. `prev_oi` (Fyers previous-day OI) and `oichp` (Fyers OI %change vs previous day) are stored only as reference-only fields, and Fyers' `oich` is deliberately not parsed/stored because the system computes its own intraday deltas in a later phase.
23. `store_snapshot()` in `market/store.py` append-inserts every strike into the `snapshots` table with columns `(ts, index_name, option_type, strike, expiry, oi, ltp, volume, prev_oi, oichp)` via `executemany`, using `chain.fetched_at` as the single shared `ts` for all rows.
24. `store_snapshot()` returns 0 and logs a warning when a chain has no strikes to store, and `store_chains()` exists as a standalone batch helper that wraps each per-index store in `try/except` recording 0 on failure (per-index resilience, spec §11), though `run_fetch()` calls `store_snapshot()` directly with its own `try/except`.
25. `snapshot_exists_since(ts)` runs `SELECT 1 FROM snapshots WHERE ts >= %s LIMIT 1` to back the `/tick` market-minute de-dup so a double-fired cron within the same minute is skipped.
26. The instrument registry in `market/instruments.py` is loaded once from the `instruments` table (selecting `name, symbol, strike_interval, lot_size, expiry_weekday, price_mult, is_active WHERE is_active ORDER BY name`) and cached in module-level `_cache`, reloadable via `all_instruments(refresh=True)`.
27. `get_instrument()` raises `KeyError` pointing to the `db/schema.sql` seed when a requested instrument name is not in the registry, making the `instruments` table the single source of truth for symbol, strike interval, lot size, expiry weekday, and Sensex price multiplier.
28. The expected Fyers underlying symbols are NSE-style values like `NSE:NIFTY50-INDEX` stored in the `instruments` table, and no `depth()` OI-frozen fallback is implemented in any Phase 1 file despite the spec mentioning it.
29. `market/fetch.py` provides a CLI `main()` (`uv run python -m market.fetch [strikecount]`) that configures logging, reads `strikecount` from `sys.argv` (default 10), prints `json.dumps(run_fetch(...))`, and always calls `close_pool()` in a `finally` block.

## Phase 2 — Expiry / DTE awareness · [spec §3]

File: `compute/expiry.py`

1. `days_to_expiry(expiry, today)` computes days-to-expiry as `(expiry - today).days`, a calendar-day count (not trading days) that yields 0 on the expiry day itself.
2. DTE is intentionally calendar days from the chain's nearest expiry to today's IST date, which the module docstring states reproduces the spec's weekday table (e.g. Wed → next-Tue ≈ 6).
3. `days_to_expiry` returns a negative integer when expiry is already in the past (today after expiry), since `(expiry - today).days` has no clamping to a lower bound.
4. When DTE is negative (a stale/past expiry), both `is_expiry_day` (`dte == 0`) and `near_expiry` (`dte == 1`) evaluate to False, so a past-expiry index is treated as neither expiry-day nor near-expiry.
5. `index_expiry(index_name, expiry, today)` constructs an `IndexExpiry` carrying `index_name`, `expiry`, the computed `dte`, an `is_expiry_day` flag, and a `near_expiry` flag.
6. The `is_expiry_day` flag is set True exactly when `dte == 0` (0-DTE).
7. The `near_expiry` flag is set True exactly when `dte == 1` (1-DTE).
8. `assess(nifty, sensex)` takes a required NIFTY `IndexExpiry` and an optional SENSEX `IndexExpiry` (which may be None) and returns an `ExpiryAssessment`.
9. A `sensex_missing` boolean is computed as `sensex is None` and is one of the conditions that triggers cross-check suppression.
10. Cross-check suppression (`suppress`) is True when NIFTY is on its expiry day (`nifty.is_expiry_day`), OR SENSEX is missing (`sensex_missing`), OR SENSEX is on its expiry day (`sensex.is_expiry_day`).
11. Because suppression short-circuits on `sensex_missing` before evaluating `sensex.is_expiry_day`, the code avoids a None-attribute error when SENSEX is absent.
12. The low-weight near-expiry flag (`low_weight`) is computed only when NOT suppressed AND (NIFTY is near-expiry OR SENSEX exists and is near-expiry), so a suppressed assessment can never also be low-weight.
13. When SENSEX is missing, the label is set to `LABEL_NIFTY_ONLY` ("NIFTY-ONLY — Sensex unavailable").
14. When suppression is active but SENSEX is present, the label is set to `LABEL_SUPPRESSED` ("EXPIRY — Sensex cross-check paused").
15. When not suppressed but `low_weight` is True, the label is set to `LABEL_LOW_WEIGHT` ("near-expiry, low weight").
16. When neither suppressed nor low-weight, the label is set to `LABEL_ACTIVE` ("cross-check active").
17. Label selection is an ordered if/elif/else chain checking `sensex_missing` first, then `suppress`, then `low_weight`, then the active default, so missing-SENSEX takes label priority over the generic suppression label even though both set `suppress` True.
18. The returned `ExpiryAssessment` carries `nifty`, `sensex` (or None), the boolean `suppress_cross_check`, the boolean `low_weight`, and the chosen `label`.
19. The four banner labels are module-level string constants (`LABEL_SUPPRESSED`, `LABEL_NIFTY_ONLY`, `LABEL_LOW_WEIGHT`, `LABEL_ACTIVE`) kept centralized for tuning and test reference.
20. `assess_from_dates(nifty_expiry, sensex_expiry, today)` builds the NIFTY `IndexExpiry` with hardcoded name `"NIFTY"` and, only if `sensex_expiry` is truthy, the SENSEX `IndexExpiry` with hardcoded name `"SENSEX"`, otherwise passes None (and because a `date` object is always truthy, only the None case yields `sensex=None`, propagating into the missing-SENSEX / NIFTY-ONLY path).
21. The code implements only DTE computation, weekday-table-equivalent calendar dating, 0-DTE suppression, and 1-DTE low-weight flagging; it does NOT implement explicit weekday-name tagging (Tue/Thu strings) nor per-weekday/DTE bucketing — those live in Phase 5.

## Phase 3 — Strike auto-selection · [spec §4]

Files: `market/selection.py`, `market/zones.py`, `compute/strikes.py`, `schemas/zones.py`

1. A `Zone` is a Nifty price band with a role of `RESISTANCE` or `SUPPORT`, validated in `schemas/zones.py` such that `low <= high` (the ordered `model_validator` raises "zone low must be <= high" otherwise).
2. `Zone.option_type` maps `RESISTANCE → "CE"` (track call OI) and `SUPPORT → "PE"` (track put OI) via a property.
3. `Zone.from_level(role, level)` converts a single price level `L` into the band `[L - SINGLE_LEVEL_PAD, L + SINGLE_LEVEL_PAD]` where `SINGLE_LEVEL_PAD = 25`, implementing spec §4's "single level L → [L-25, L+25]".
4. `Zone.pair_from_bands(band_a, band_b)` takes two unlabeled `(low, high)` bands and assigns roles by midpoint — the higher midpoint becomes RESISTANCE, the lower becomes SUPPORT — so callers can pass raw bands without labeling.
5. `set_zones()` in `market/zones.py` upserts the day's zones into the `zones` table (`trading_date, role, low, high`) with `ON CONFLICT (trading_date, role) DO UPDATE`, making zones re-settable for the same day.
6. `set_zones()` enforces the spec contract that when both roles are present the RESISTANCE band must sit strictly above the SUPPORT band, raising `ValueError` ("roles look swapped") if `res.low <= sup.high`.
7. `get_zones()` reads the `zones` table for a trading_date and returns `Zone` objects `ORDER BY low DESC`, so the highest band (RESISTANCE) comes before the lower band (SUPPORT).
8. `candidate_strikes(low, high, interval)` in `compute/strikes.py` computes strikes that are multiples of `interval` within `[low-interval, high+interval]` using `lo = ceil((low-interval)/interval)*interval` and `hi = floor((high+interval)/interval)*interval`, returning an empty list if `hi < lo`.
9. For Nifty the interval is the instrument's `strike_interval` (multiples of 50, candidates in `[low-50, high+50]`) and for Sensex it is multiples of 100, both sourced from the instrument's `strike_interval` field rather than hardcoded in this phase.
10. `map_band(low, high, price_mult)` scales a Nifty band onto another index by multiplying both endpoints by `price_mult` (Sensex ≈ Nifty × 3.20), and `select_zone_walls()` applies it only when `price_mult != 1.0` (Nifty leaves the band unchanged at 1.0).
11. `select_wall()` builds an `oi_by_strike` dict mapping `strike → oi` only for chain strikes whose `option_type` matches the zone's type AND whose strike is in the candidate set, returning None if no such candidate strike exists in the chain (e.g. zone lies outside the fetched strikecount).
12. The WALL is chosen as the highest-OI candidate strike via `max(oi_by_strike, key=lambda k: (oi_by_strike[k], -abs(k - centre)))` where `centre = (low + high) / 2`, so ties on OI break toward the strike nearest the band centre for deterministic selection.
13. The MONITORED set is computed as `[wall - interval, wall, wall + interval]` (three strikes, ascending) and stored on `WallSelection.monitored`.
14. `select_wall()` records `band_low = min(candidates)`, `band_high = max(candidates)` (the candidate scan extent), `wall_oi` (the wall strike's OI at lock time), and persists the `interval` field on the returned `WallSelection` so the locked interval travels with the selection.
15. `select_zone_walls(zone, chain, interval, price_mult)` is the per-zone-per-index entry point that maps the band (for Sensex) then calls `select_wall` using `chain.strikes`, `zone.option_type`, the chain's `index_name`, and the chain's `expiry`.
16. `plan_locks(zones, chains, instruments, already_locked)` is a PURE function (no DB) that iterates every zone × every chain index, skips indices not present in `instruments`, looks up each `Instrument` via `instruments.get(name)`, passes both `inst.strike_interval` and `inst.price_mult` to `select_zone_walls`, and skips any `(role, index_name, expiry)` already in `already_locked` so a same-day re-run is a no-op.
17. `plan_locks()` handles expiry-roll by keying on `(role, index_name, chain.expiry)`: a rolled expiry has no matching already-locked key, so it is selected and locked fresh on the new chain.
18. `lock_walls()` in `market/selection.py` orchestrates locking by calling `get_zones()`, loading `all_instruments()`, reading `get_locked_keys()`, calling `plan_locks()`, and persisting newly planned `WallSelection`s into the `monitored_strikes` table.
19. `lock_walls()` returns early with an empty list and a warning if no zones are set for the trading_date, and returns an empty list with an info log if `plan_locks` produces nothing new (walls already locked).
20. `get_locked_keys(trading_date)` returns the set of `(role, index_name, expiry)` tuples already in `monitored_strikes` for that day, used to avoid re-locking.
21. `_insert_lock()` persists a lock into `monitored_strikes` (`trading_date, role, index_name, option_type, expiry, wall_strike, monitored, wall_oi_at_lock`) with `INSERT ... ON CONFLICT (trading_date, role, index_name, expiry) DO NOTHING` and returns `cur.rowcount` (1 = inserted, 0 = already locked) so concurrent lockers are race-safe with first-one-wins.
22. `lock_walls()` only appends to its returned `locked` list the rows `_insert_lock` actually inserted (rowcount 1), so the returned list is accurate under concurrent races, and each returned dict has keys `role, index, option_type, wall, monitored, expiry` (expiry serialized via `str(sel.expiry)`).
23. `_warn_unlockable()` logs a warning for any `(role, index, expiry)` for which a chain exists and the index is a known instrument but which is not in the coverable set (already-locked plus newly-planned), surfacing walls that fell outside the fetched strikecount instead of failing silently (spec §11).
24. `check_migration(sel, current_oi)` flags drift without moving the wall: it reads the locked wall's current OI plus the OI at `wall - sel.interval` and `wall + sel.interval`, picks the larger-OI neighbor (preferring "up" when above >= below, else "down"), and returns a `MigrationFlag` with `shifting=True` only when that neighbor's OI strictly exceeds the wall's OI, otherwise `shifting=False` with detail "wall holding".
25. When `check_migration` detects a shift the `MigrationFlag` carries `from_strike=wall_strike`, `to_strike=`the larger neighbor, `direction` ("up"/"down"), and a detail string, and the function never moves or re-picks the wall — the caller acts on the flag (spec §4: lock; flag migration, never re-pick).
26. Missing OI values in `check_migration` default to 0 via `current_oi.get(strike, 0)`, so absent strikes are treated as zero OI.
27. `lock_session(trading_date, strikecount=10)` is the convenience entry that obtains a Fyers client (returning `[]` and logging an error if auth fails), fetches both chains via `fetch_chains`, and delegates to `lock_walls`.
28. `SetZonesRequest` (the Phase 6 body) requires exactly two zones (raising "exactly two zones are required" otherwise) and each `ZoneBand` must have positive bounds and `low <= high`, with the API assigning roles via `pair_from_bands` rather than the client labeling them.

## Phase 4 — Verdict engine (the brain) · [spec §5]

Files: `config/thresholds.py`, `compute/verdict.py`, `compute/series.py`, `compute/engine.py`, `schemas/verdict.py`

1. `window_change()` computes per-strike Δ% as `(latest_oi - baseline_oi) / baseline_oi * 100.0`, where `latest_oi = series[-1].oi` and `baseline_oi` is the OI of the last snapshot whose `ts <= (now - window_minutes)`.
2. `window_change()` iterates the ascending series, keeps updating baseline to each point with `p.ts <= cutoff`, and breaks at the first point past the cutoff — a STRICT `value_at_or_before(now - window)` baseline rather than approximating with a shorter window.
3. `window_change()` returns `change_pct=None` (insufficient) when the series is empty, when no snapshot exists at-or-before the cutoff, or when the baseline OI is 0, so an early-session strike reads as insufficient instead of producing a misleading Δ%.
4. `classify_magnitude()` buckets `|Δ%|` into "strong" (>= `MAG_STRONG_PCT` 20.0), "signal" (>= `MAG_SIGNAL_PCT` 10.0), "mild" (>= `MAG_MILD_PCT` 5.0), and "noise" (< 5.0), returning "noise" when `change_pct` is None.
5. `direction_of()` returns "flat" when `change_pct` is None or `|change_pct| < FLAT_PCT` (which equals `MAG_MILD_PCT = 5.0`), otherwise "up" if positive (OI building) or "down" if negative (OI unwinding) — making "noise magnitude" equivalent to "flat direction".
6. `FLAT_PCT` is set to `MAG_MILD_PCT` (5.0) in `config/thresholds.py` as the spec-derived interpretation that a move under 5% is flat/quiet, with a comment noting the JSX reference that would pin it exactly is absent so the spec wins.
7. `compute_reads_streak()` walks adjacent `(a, b)` pairs computing `pct = (b.oi - a.oi)/a.oi*100.0`, labeling each pair "up"/"down" when `|pct| >= READ_THRESHOLD_PCT` (3.0) else "flat", and labeling the pair "flat" (skipping division) when `a.oi == 0`.
8. `compute_reads_streak()` counts `reads_total` as the number of non-flat pair moves, and computes the trailing streak by walking moves in reverse from the last pair, incrementing while the direction matches the most recent move and stopping at the first mismatch.
9. `compute_reads_streak()` returns `(reads_total, 0, "flat")` when there are no moves or when the most recent pair is flat, so a flat latest pair breaks the streak to zero.
10. `strike_signal()` assembles a `StrikeSignal` with `change_pct` rounded to 2 decimals (or None), `direction` from `direction_of()`, `magnitude` from `classify_magnitude()`, `reads`/`streak` from `compute_reads_streak()`, `trend = (streak >= TREND_STREAK = 3)`, and `insufficient = (change_pct is None)`.
11. `strike_signal()` populates `oi_latest`/`oi_baseline` directly from `window_change()`'s returned latest/baseline, so a strike with `change_pct=None` (e.g. baseline OI 0) can still carry non-None `oi_latest`/`oi_baseline`, and it stamps identity fields `index_name, option_type, strike, expiry, is_wall` onto every `StrikeSignal`.
12. `index_state()` maps a wall's `StrikeSignal` to "nodata" when insufficient, "flat" when direction is flat, "building" when direction is up, and "unwinding" when direction is down.
13. `build_wall_signal()` locks the wall as `signals_by_strike[ms.wall_strike]`, collects neighbors from `ms.monitored` (excluding the wall and only including strikes actually present in `signals_by_strike`, so a monitored strike with no computed signal is silently dropped), and builds a summary string `"{state}/{wall.magnitude}/streak{wall.streak}"` — never silently re-picking the wall.
14. `build_wall_signal()` flags neighbor migration as context only: coalescing missing `oi_latest` to 0, it finds the neighbor with the largest `oi_latest` exceeding the wall's `oi_latest` and, if one exists, sets migration to `"OI peak shifting {up|down}: {wall_strike} → {overtaker.strike}"` (Unicode `→`, "up" when `overtaker.strike > wall_strike`).
15. `zone_verdict()` applies §3 expiry suppression FIRST: `suppressed` is true when `assessment.suppress_cross_check` is set OR the Sensex `WallSignal` is None, and a suppressed zone always emits a NIFTY-ONLY row with `sensex=None` and `suppressed=True`.
16. In the suppressed NIFTY-ONLY branch: building Nifty → `"{hold} (NIFTY-ONLY)"` / MODERATE; unwinding Nifty → `"{brk} (NIFTY-ONLY)"` / LOW when `nifty.wall.streak >= MIN_CONFIRM_STREAK` (2) else `"{brk} (NIFTY-ONLY, UNCONFIRMED)"` / UNCONFIRMED; flat → "NO SIGNAL (NIFTY-ONLY)" / NONE; nodata → "NO SIGNAL (NIFTY-ONLY)" / NONE with an "insufficient history" meaning.
17. `_labels()` picks `("CAP HOLDING", "BREAKOUT")` for role RESISTANCE and `("FLOOR HOLDING", "BREAKDOWN")` for role SUPPORT, and `zone_verdict()` sets `option_type` to "CE" for RESISTANCE and "PE" for SUPPORT.
18. When both indices are moving and both building, conviction is HIGH if (both walls' magnitude is at least "signal" via `magnitude_at_least`) OR (either `wall.trend` is true), else MODERATE, with `verdict =` the hold label.
19. When both indices are moving and both unwinding, the move is "confirmed" only when `nifty.wall.streak >= 2` AND `sensex.wall.streak >= 2` (both-streak rule), giving `verdict =` the break label with conviction HIGH if either `wall.trend` else MODERATE; otherwise conviction is UNCONFIRMED with the meaning that a single read / one unwind is a fake.
20. When both indices are moving but in opposite directions, the verdict is "DIVERGENCE" / LOW with a fake-out-risk meaning.
21. When exactly one index is moving (`_moving(n) != _moving(s)`, with `_moving = state in {building, unwinding}`), the verdict is "PARTIAL — one index only" / LOW; when both are quiet (flat or nodata), the verdict is "NO SIGNAL" / NONE.
22. `zone_verdict()` derives `dte_n = assessment.nifty.dte` and `dte_s = assessment.sensex.dte if assessment.sensex else None`, and passes `assessment.label` through to `ZoneVerdict.expiry_label`, on every row in both the suppressed and non-suppressed branches.
23. `_index_wall()` in `compute/engine.py` pulls each monitored strike's series via `read_oi_series` since `(now - SERIES_LOOKBACK_MINUTES = 120 min)`, scores every strike in `ms.monitored` with `strike_signal()` (using the same `ms.expiry`/`ms.option_type`) flagging `is_wall = (strike == ms.wall_strike)`, and feeds the dict to `build_wall_signal()`.
24. `build_state()` defaults `now` to `datetime.now(IST)` (UTC+5:30), converts the incoming now to IST via `now.astimezone(IST)` (using `ist_now` for trading_date defaulting, weekday `"%a"`, and the `assess_from_dates` as-of date while `VerdictState.ts` stores the original now), falls back `window_minutes` to `DEFAULT_WINDOW_MINUTES` (15) when not in `WINDOW_CHOICES` (15, 30), and only defaults `trading_date` to `ist_now.date()` when it is None.
25. `build_state()` returns early `VerdictState`s with notes (reusing a shared base dict of ts/trading_date/weekday/window_minutes) when no monitored strikes exist or when no NIFTY walls are locked, treating NIFTY as the primary anchor index.
26. `build_state()` determines `nifty_exp`/`sensex_exp` as the first-seen expiry per index across zones, calls `assess_from_dates(nifty_exp, sensex_exp, ist_now.date())`, groups monitored strikes by role then index, iterates zones sorted by `_ROLE_ORDER` (RESISTANCE=0 before SUPPORT=1), skips any zone lacking a NIFTY wall with a warning, and builds the Sensex wall only when a SENSEX `MonitoredStrike` exists for that zone.
27. `read_monitored_strikes()` selects all rows from `monitored_strikes` for the trading_date ordered by role, index_name; `read_oi_series()` selects `(ts, oi)` from `snapshots` filtered by index_name/option_type/strike/expiry with `ts >= since` ordered ascending by ts, returning `SnapshotPoint` objects (the ascending order satisfied by walking `snapshots_series_idx` forward).
28. The `/state` contract types live in `schemas/verdict.py` as `VerdictState` (ts, trading_date, weekday, window_minutes, expiry, zones, note) containing `ZoneVerdict` rows (role, option_type, verdict, conviction, meaning, nifty_sig, sensex_sig, dte_n, dte_s, suppressed, expiry_label, nifty, sensex) each wrapping `WallSignal` (state, wall, neighbors, summary, migration) and `StrikeSignal` (oi_latest, oi_baseline, change_pct, direction, magnitude, reads, streak, trend, insufficient), with `Conviction` constrained to HIGH/MODERATE/LOW/UNCONFIRMED/NONE.
29. All tunable thresholds live in one config block in `config/thresholds.py` — `DEFAULT_WINDOW_MINUTES=15`, `WINDOW_CHOICES=(15,30)`, `SERIES_LOOKBACK_MINUTES=120`, `READ_THRESHOLD_PCT=3.0`, `TREND_STREAK=3`, `MIN_CONFIRM_STREAK=2`, `MAG_MILD_PCT=5.0`, `MAG_SIGNAL_PCT=10.0`, `MAG_STRONG_PCT=20.0`, `FLAT_PCT=5.0` — plus the helper `magnitude_at_least()` comparing labels against `MAGNITUDE_ORDER` ("noise", "mild", "signal", "strong").
30. `main()` is a CLI entry that parses `argv[1]` as an ISO trading_date and `argv[2]` as the window, runs `build_state()`, prints `state.model_dump(mode="json")` as indented JSON, and calls `close_pool()` in a `finally` block.

## Phase 5 — Persist verdicts (backtest dataset) · [spec §1, §11]

Files: `market/verdict_store.py`, `compute/persist.py`, `market/tick.py`

1. The `verdicts` table has exactly 15 writable columns inserted in a fixed order via the `_INSERT` statement in `market/verdict_store.py`: ts, trading_date, weekday, window_minutes, role, option_type, verdict, conviction, meaning, nifty_sig, sensex_sig, dte_n, dte_s, suppressed, expiry_label.
2. `insert_verdicts(rows)` bulk-inserts via `cur.executemany` and returns the count of rows written, building each value tuple in the same fixed column order as `_INSERT`.
3. In `insert_verdicts`, required keys (ts, trading_date, weekday, window_minutes, role, option_type, verdict, conviction) are read directly with `r[...]` (raising KeyError if missing), while nullable keys on the read/insert surface (meaning, nifty_sig, sensex_sig, dte_n, dte_s, expiry_label) are read via `r.get(...)` so a missing key becomes NULL — though the persist path always supplies `meaning` from the required `ZoneVerdict.meaning`.
4. In `insert_verdicts`, the `suppressed` value defaults to `False` via `r.get("suppressed", False)` when the key is absent.
5. `insert_verdicts` is append-only with no `ON CONFLICT` clause, so it never de-duplicates at the SQL layer (tick-level de-dup is deferred to Phase 6).
6. If `insert_verdicts` is passed an empty list it logs a warning "no verdict rows to insert" and returns 0 without opening a connection; on a write it logs "inserted %d verdict rows" at INFO and opens a pooled connection only when called (via `get_conn()`), never at import time.
7. `read_verdicts(trading_date)` selects all columns WHERE `trading_date =` the given date, ordered `ts DESC, role` (newest tick first then by role), mapping each result dict into a `VerdictRecord`.
8. `read_verdicts_range(start, end, role=None)` selects rows WHERE `trading_date BETWEEN start AND end` inclusive, ordered `trading_date DESC, ts DESC, role`, returning a list of `VerdictRecord`, and adds an `AND role = %s` filter when a non-None `role` is supplied to scope the result to one zone side.
9. `bucket_counts(start, end, role=None)` runs a SQL `GROUP BY weekday, verdict` with `count(*) AS n` over the trading_date range to produce per-weekday verdict tallies for the spec §3/§11 per-weekday judging, adding an `AND role = %s` filter when a non-None role is given so buckets line up with a role-filtered record list.
10. `bucket_counts` aggregates the SQL rows in Python into a `by_weekday` dict mapping each weekday to `{"count": total, "by_verdict": {verdict: n}}`, summing per-verdict counts into the weekday total.
11. `bucket_counts` returns a list of dicts shaped like `BucketStat` with keys `bucket`, `count`, `by_verdict`, emitted in the fixed `_WEEKDAY_ORDER` of Mon, Tue, Wed, Thu, Fri, Sat, Sun, and only includes weekdays that actually appear in the data (`if wd in by_weekday`).
12. The `weekday` column stores the literal three-letter strings from `_WEEKDAY_ORDER` (Mon..Sun), the fixed ordering used for bucketing.
13. `read_verdicts`, `read_verdicts_range`, and `bucket_counts` all rely on the cursor returning dict-shaped rows (accessed as `r["weekday"]`, `r["verdict"]`, `r["n"]` and splatted via `VerdictRecord(**r)`).
14. `_row(state, z)` in `compute/persist.py` maps one `ZoneVerdict` plus its `VerdictState` envelope into the 15-column row dict, pulling ts/trading_date/weekday/window_minutes from the state and role/option_type/verdict/conviction/meaning/nifty_sig/sensex_sig/dte_n/dte_s/suppressed/expiry_label from the zone.
15. `persist_state(state)` logs one verdict row per zone by building `[_row(state, z) for z in state.zones]` and calling `insert_verdicts`, returning the number of rows written.
16. `persist_state` returns 0 early (logging "no zones in state ... nothing to persist" at INFO) when `state.zones` is empty, e.g. when walls have not been locked yet.
17. `persist_state` deliberately logs EVERY zone including quiet NO-SIGNAL and suppressed NIFTY-ONLY rows, because the backtest needs the full series and the `suppressed` flag plus weekday let distorted expiry days be bucketed out later.
18. `build_history(start, end, role=None)` assembles the `/history` payload by combining `read_verdicts_range(start, end, role)` records with `bucket_counts(start, end, role)` mapped into `BucketStat`s, returning a `VerdictHistory(start=start, end=end, records=records, weekday_buckets=buckets)`.
19. `build_history` passes the same `role` to both the records read and the bucket read so a role-scoped view (e.g. RESISTANCE-only) shows matching role-scoped records and weekday counts with no mismatch.
20. `run_tick` executes the unified cycle in strict order: get Fyers client → fetch both chains → store snapshots → lock walls from the SAME chains → `build_state` → persist verdicts.
21. `run_tick` closes the Phase-4 FN-1 gap by feeding a SINGLE Fyers fetch (`fetch_chains`) into both `store_chains` (snapshots) and `lock_walls`, guaranteeing the persisted OI series covers the locked/monitored strikes.
22. `run_tick` stores the WHOLE chain (not just monitored strikes) via `store_chains(chains)` into `summary["snapshots"]` (before the compute/persist try block) so the extra strikes provide history for wall-migration (spec §4) and expiry re-lock.
23. `run_tick` never raises: every stage degrades into a summary dict so a scheduler ping always gets an answer, and snapshots already stored are retained in the summary even if the later compute/persist step fails.
24. `run_tick` defaults `trading_date` to today in IST (`datetime.now(IST).date()`, IST = UTC+5:30), and defaults `strikecount=10` and `window_minutes=DEFAULT_WINDOW_MINUTES`.
25. `run_tick` returns early with `{"trading_date": ..., "error": "auth: ..."}` if `get_fyers_client()` raises, and with `{"trading_date": ..., "error": "no chains fetched"}` if `fetch_chains` returns falsy, skipping storage/compute/persist in both cases.
26. `run_tick` wraps the lock/compute/persist block (steps 2–4) in a `try/except` that on failure logs the exception and sets `summary["error"] = "compute/persist: ..."`, so a DB hiccup never discards the already-stored snapshots.
27. `run_tick` calls `lock_walls(td, chains)` on EVERY tick rather than once at session start, relying on `lock_walls` idempotency (plan_locks skips locked keys; `INSERT ... ON CONFLICT`) to re-attempt zones set late or walls that just came into range.
28. On the success path `run_tick` updates the summary with `locked`, `verdicts_written`, a `zones` list of `{role, verdict, conviction}` per zone, and `state.note`.
29. `run_tick` lazily imports `get_fyers_client` inside the function to avoid importing the Fyers SDK at module import time.
30. `run_tick` is documented as a LIVE same-day cycle because `build_state` windows the OI series off *now*; passing a historical trading_date won't replay that day (Fyers can't re-serve past intraday chains) and exists only for same-day overrides/tests.
31. `is_market_hours(now=None)` normalizes a passed-in `now` to IST via `.astimezone(IST)`, then returns True only on weekdays (`ist.weekday() < 5`) when the IST time is between `MARKET_OPEN` 09:15 and `MARKET_CLOSE` 15:30 inclusive, a defensive re-check of the cron window.
32. `already_ticked_this_minute(now=None)` implements M5-2 de-dup by normalizing `now` to UTC (`.astimezone(timezone.utc)`), computing the current UTC clock-minute start (`replace(second=0, microsecond=0)`), and returning `snapshot_exists_since(minute_start)`, so a double-fired cron in the same minute is skipped to avoid duplicate snapshot+verdict rows that would skew backtest counts.
33. The docstrings of `is_market_hours` and `run_tick` reference a `force=` override, but neither function actually defines a `force` parameter — the override lives at the Phase-6 `GET /tick` endpoint (`?force=true`).
34. The `main()` CLI entrypoint configures logging via `logging.basicConfig` at INFO with format `"%(asctime)s %(levelname)s %(name)s: %(message)s"`, parses positional argv (argv[1] trading_date ISO, argv[2] strikecount default 10, argv[3] window default `DEFAULT_WINDOW_MINUTES`, each guarded by a `len(sys.argv)` check), prints `run_tick(...)` as indented JSON with `default=str`, and always calls `close_pool()` in a `finally` block.
